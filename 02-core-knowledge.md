# 02 核心知识底稿

## 1. AI Infra 到底是什么

AI Infrastructure 是支撑 AI 系统从数据准备、模型训练、模型部署到生产监控的整套底层能力。它不是某个框架，也不是某台 GPU 服务器，而是“算力 + 数据 + 编排 + 运营”的组合。

一句更实用的话：

**AI Infra 的目标，是让模型能被稳定、可扩展、可治理、可复现地训练和运行。**

## 2. 计算层：为什么 GPU 是核心

### GPU 的价值

深度学习的核心是大规模矩阵运算，GPU 擅长并行处理，所以在训练和大规模推理里几乎是默认选项。

### CPU 仍然重要

CPU 不是没用，而是主要承担：

- 数据预处理
- 轻量推理
- 调度与控制逻辑
- 日志、监控、队列、元数据管理

### TPU / 专用加速器

TPU、DPU、ASIC 代表的是“更垂直的优化”。它们通常在大规模云环境或特定场景里更强，但生态和通用性不如 GPU。

### 你真正要关心的不是型号，而是指标

- 显存是否够
- GPU 间通信是否够快
- GPU 利用率是否高
- 单位训练成本是否合理
- 推理时每个请求的成本是否可接受

## 3. 存储层：AI 项目常常先被 I/O 卡住

AI 系统不是只有模型参数，真正占空间的通常还有：

- 原始训练数据
- 清洗后的中间数据
- 特征
- checkpoint
- 日志与指标
- 多个模型版本

### 三类常见存储

**对象存储**

适合放大规模训练数据、模型产物、长期归档。优点是便宜、可扩展、耐久；缺点是延迟较高。

**块存储 / 本地 NVMe**

适合高吞吐读写、训练缓存、热点 checkpoint。优点是快；缺点是贵且容量有限。

**分布式文件系统**

适合大集群共享数据。重点不是“会不会配”，而是理解它解决了单机磁盘无法扩展的问题。

### 你要形成的直觉

很多训练慢，不是模型算得慢，而是数据喂不进去。

## 4. 网络层：分布式训练的隐形天花板

单机训练主要看本机算力，多机训练很快就会遇到网络瓶颈。

### 网络决定什么

- 梯度同步速度
- 多机扩展效率
- checkpoint 回写速度
- 训练吞吐是否线性增长

### 关键概念

- **NCCL**：GPU 通信库
- **AllReduce**：分布式训练常见梯度同步方式
- **NVLink**：单机多 GPU 高速互联
- **InfiniBand / RDMA / RoCE**：多机高性能网络

### 一个重要判断

如果 GPU 数量增加了，但训练速度没有接近同比例增长，通常就要开始怀疑通信、数据加载或同步策略。

## 5. 训练系统：把“能训练”变成“能规模化训练”

### 训练系统真正解决的事

- 多卡/多机资源协调
- 数据并行、模型并行、流水线并行
- 失败后的恢复
- 实验可复现
- 长时间作业的稳定性

### 三种常见并行思路

**数据并行**

每张卡拿不同 batch，最容易理解，也最常见。

**模型并行**

模型太大，单卡放不下，就把模型拆到多卡。

**流水线并行**

按层或阶段切分计算，让不同设备分担不同阶段。

### 训练系统里更重要的指标

- time-to-accuracy
- GPU utilization
- samples/sec
- checkpoint 成功率
- 恢复时间

只盯 FLOPS 没意义，能稳定把训练跑完更重要。

## 6. 推理系统：训练完不等于可上线

训练系统解决“把模型学出来”，推理系统解决“把模型稳定服务出去”。

### 推理系统更关心的指标

- p50 / p95 / p99 延迟
- 每秒请求数
- 并发容量
- 错误率
- 单请求成本

### 典型优化手段

- dynamic batching
- 模型量化
- TensorRT / ONNX Runtime / Triton 这类运行时优化
- 缓存
- autoscaling
- canary / blue-green 发布

### 训练和推理的根本差异

训练要的是“更快学完”，推理要的是“稳定地、持续地、便宜地服务请求”。

## 7. 数据基础设施：很多 AI 系统本质上是数据系统

模型质量很大程度上取决于数据质量、数据时效性和数据可追踪性。

### 常见组件

- 数据湖 / 数据仓库
- ETL / ELT / 流处理
- 特征存储
- 向量数据库
- 数据版本控制

### 特征存储的价值

特征存储不是“多一个数据库”，而是解决训练与线上推理使用同一套特征定义的问题，减少训练-服务偏差。

## 8. MLOps：让 AI 从“实验”走向“工程”

MLOps 不只是自动化脚本，它解决的是：

- 实验跟踪
- 模型版本管理
- 数据版本管理
- 流水线自动化
- 审批与发布流程
- 复现实验结果

### 你应该先掌握的最小组合

- `MLflow`：实验、参数、指标、模型版本
- `DVC`：数据版本与 pipeline
- `Kubeflow` 或 `Ray`：当你需要更强的平台化能力时再上

## 9. 生产运营：真正难的是上线之后

线上 AI 系统会遇到的不是“模型代码报错”这么简单，而是：

- 漂移
- 资源打满
- 成本失控
- 延迟波动
- 数据质量下降
- 权限与审计问题

### 生产必须监控的几类指标

- 系统指标：CPU、GPU、显存、网络、磁盘
- 服务指标：QPS、延迟、错误率、队列长度
- 模型指标：准确率、召回率、漂移、输出分布
- 业务指标：转化率、命中率、用户反馈
- 成本指标：每小时 GPU 成本、每次训练成本、每请求成本

## 10. 部署模式不是技术问题，而是权衡问题

### 云上

优点是启动快、弹性强、试错成本低。

缺点是长期成本高、供应商绑定强、数据合规可能复杂。

### 本地

优点是控制力强、长期重负载可能更划算。

缺点是前期投入大、运维复杂、扩容慢。

### 混合

很常见，因为企业往往既想要云上的弹性，也想保留本地的控制权和敏感数据隔离。

## 11. 你最该记住的 8 条判断

1. 训练系统和推理系统要分开思考。
2. GPU 不等于 AI Infra，全链路才是。
3. 数据吞吐和网络带宽经常比模型代码更先成为瓶颈。
4. 单机跑得通，不代表能多机扩展。
5. 模型服务上线后，监控和回滚能力与模型本身同样重要。
6. MLOps 的核心是可复现和可运营，不是“平台很大”。
7. 企业落地最常见的约束不是算法，而是治理、安全、成本和人才。
8. 学 AI Infra 的最好方式，是一边学概念，一边用最小项目把链路串起来。
