# AI Infra 直读学习手册

这份手册是“直接阅读版”。

目标很简单：你打开之后就能一路读下去，把 AI Infra 的核心知识、代码骨架、真实系统案例和学习路径一次性串起来，而不是在一堆链接之间来回跳。

---

## 第 1 章：先搞清楚 AI Infra 到底是什么

很多人第一次听到 AI Infrastructure，会先想到 GPU 服务器、云厂商或者 Kubernetes。它们都有关，但都不等于 AI Infra 本身。

更准确地说，**AI Infra 是支撑 AI 系统从数据准备、模型训练、模型部署到生产监控的整套底层系统能力。**

如果把一个 AI 产品拆开看，它大概会经历这样一条链：

`数据采集 -> 数据处理 -> 模型训练 -> 模型产物保存 -> 模型上线服务 -> 监控反馈 -> 再训练`

AI Infra 的作用，就是保证这条链能稳定跑起来，而且随着规模增大依然可扩展、可复现、可治理、可控制成本。

### 为什么 AI Infra 会变成一个独立问题

因为现代 AI 工作负载和传统 Web 应用很不一样。

传统应用更像：

- CPU 为主
- 数据量可预测
- 请求模式相对稳定
- 扩容更多靠加机器和缓存

AI 应用则经常变成：

- GPU/TPU 为主
- 数据量极大
- 训练和推理的负载模型完全不同
- 算力、存储、网络都可能成为瓶颈
- 线上不仅要看接口状态，还要看模型效果、漂移和成本

所以 AI Infra 学的不是单点技术，而是系统协同。

## 第 2 章：AI Infra 的 7 层结构

你可以把 AI Infra 粗略理解成 7 层。

### 第 1 层：计算层

这里是 CPU、GPU、TPU、DPU、显存、CUDA、NVLink 这些东西所在的位置。

这一层解决的问题是：

- 模型算得够不够快
- 显存放不放得下
- 多卡能不能协同
- 单位训练成本高不高

### 第 2 层：存储层

这里放的是训练数据、checkpoint、模型产物、日志、特征和 embedding。

这一层解决的问题是：

- 数据能不能喂得够快
- checkpoint 能不能安全保存
- 数据版本能不能追踪
- 训练和推理是否使用同一套数据语义

### 第 3 层：网络层

多 GPU、多机器训练时，网络直接决定扩展效率。

这一层解决的问题是：

- 梯度同步是否足够快
- GPU 之间通信是否高效
- 存储到计算节点的数据流是否顺畅
- 推理流量能否稳定路由

### 第 4 层：训练层

这里是训练框架和分布式训练能力，如 PyTorch、TensorFlow、JAX、Horovod、Ray、DeepSpeed。

这一层解决的问题是：

- 模型怎么被训练起来
- 单卡不够时如何扩展
- 失败后如何恢复
- 如何提高吞吐和利用率

### 第 5 层：推理层

这里是 TensorFlow Serving、TorchServe、Triton、ONNX Runtime、KServe 这些服务框架所在的位置。

这一层解决的问题是：

- 模型如何以 API 形式服务请求
- 如何控制延迟和吞吐
- 如何做 batching、缓存和扩缩容
- 如何做灰度发布和回滚

### 第 6 层：MLOps / LLMOps 层

这里是 MLflow、DVC、Kubeflow、Airflow、Feast、LakeFS 等工具常出现的位置。

这一层解决的问题是：

- 实验怎么记录
- 数据和模型怎么做版本管理
- 工作流怎么自动化
- 上线流程怎么工程化

### 第 7 层：运营与治理层

这里包括监控、日志、安全、权限、审计、合规、成本控制、容量规划。

这一层解决的问题是：

- 线上模型是否稳定
- 是否存在 drift
- 是否超预算
- 谁有权限访问哪些数据和模型
- 出现问题时能不能快速定位和回滚

## 第 3 章：为什么 GPU 在 AI Infra 里这么重要

深度学习的本质是大量矩阵运算，而 GPU 擅长大规模并行计算，所以它成为现代 AI 系统的核心算力。

### 你真正要理解的不是“GPU 很强”，而是“为什么强”

一个神经网络在前向和反向传播里会做大量矩阵乘法。CPU 核心少，适合通用控制逻辑；GPU 核心多，更适合这种高度并行的计算。

所以在 AI 场景里：

- CPU 负责调度、预处理、轻量任务
- GPU 负责主要训练和大规模推理

### 但 CPU 没有消失

很多初学者会误以为 AI Infra 只有 GPU。其实不是。

CPU 依然承担：

- 数据加载与预处理
- 日志和监控代理
- 调度器和控制面
- 轻量级模型服务
- 元数据服务

真正成熟的 AI Infra，不是“全用 GPU”，而是让 CPU 和 GPU 各司其职。

### GPU 相关的几个常见判断

1. 显存不够，模型可能根本跑不起来。
2. 显存够，但通信慢，多卡扩展也可能很差。
3. GPU 很强，但数据喂不进去，训练仍然会慢。
4. 推理时不一定追求最大吞吐，有时更看重低延迟和成本。

## 第 4 章：存储不是配角，很多系统先死在 I/O

AI 项目的存储需求非常重，因为真正占空间的通常不是代码，而是数据和中间产物。

一个正常项目里会有：

- 原始训练数据
- 清洗后的训练数据
- feature / embedding
- 训练过程中的 checkpoint
- 模型导出文件
- 监控日志
- 多个实验版本

### 三类最常见的存储

#### 1. 对象存储

适合：

- 放大规模训练数据
- 放模型产物
- 长期保存 checkpoint

优点：

- 容量大
- 成本低
- 易扩展

缺点：

- 延迟比本地盘高
- 不适合高频随机写

#### 2. 本地 NVMe / 块存储

适合：

- 高吞吐训练缓存
- 热数据
- 快速 checkpoint 中转

优点是快，缺点是贵、容量有限。

#### 3. 分布式文件系统

适合：

- 多机共享数据
- 大规模集群训练
- 统一数据路径

学习重点不是“背名字”，而是理解：**当单机磁盘不够用时，数据层必须和计算层一起扩展。**

### 你要形成的关键直觉

如果 GPU 利用率很低，不要只怪模型代码，很可能是数据管道或存储吞吐有问题。

## 第 5 章：网络为什么会决定大模型训练能不能扩展

单卡训练时，你主要关心算力和显存。多卡、多机之后，网络开始成为隐藏上限。

### 分布式训练为什么需要高速网络

因为每次反向传播之后，多张卡通常需要同步梯度。如果同步太慢，GPU 就会等网络，算力浪费非常严重。

### 几个最重要的概念

- `AllReduce`：多卡之间同步梯度的常见方式
- `NCCL`：NVIDIA 的 GPU 通信库
- `NVLink`：单机多 GPU 高速互联
- `InfiniBand / RDMA / RoCE`：多机高性能网络

### 你该如何理解扩展效率

假设单卡训练要 8 小时。

- 理想情况下，8 卡训练接近 1 小时
- 现实中，可能是 1.2 小时、1.5 小时，甚至更差

差的那部分，往往来自：

- 梯度同步开销
- 数据加载瓶颈
- 参数更新阻塞
- 节点间带宽不足

所以 AI Infra 不只是“堆卡”，而是要让卡之间协同高效。

## 第 6 章：训练系统关注的是“怎么把模型稳定训出来”

很多人会把训练理解为“写个 for loop 然后跑”。这在学习阶段没问题，但在真正的 AI Infra 里，训练系统关心的事情更多。

### 训练系统真正要解决的 5 个问题

1. 如何把任务分发到多卡或多机
2. 如何同步梯度和参数
3. 如何在失败时恢复
4. 如何把吞吐和资源利用率尽量做高
5. 如何让实验结果可追踪、可复现

### 三种最常见的并行方式

#### 数据并行

每张卡处理不同 batch，最后同步梯度。

这是最常见、最容易理解的方式。

#### 模型并行

模型太大放不下一张卡时，把模型切到多张卡。

#### 流水线并行

把模型不同阶段放在不同设备上，让计算像流水线一样推进。

### 一个你一定要记住的事实

训练系统优化的目标，不是单纯让 loss 降得更快，而是让系统层面更高效，例如：

- 每秒处理更多样本
- 用更少资源达到相同精度
- 训练失败后能快速恢复
- 多卡扩展损耗更小

## 第 7 章：最小训练代码应该长什么样

你至少应该能一眼看懂一个单 GPU 训练循环。

```python
import torch
from torch import nn
from torch.utils.data import DataLoader, TensorDataset

x = torch.randn(1024, 128)
y = torch.randint(0, 10, (1024,))
loader = DataLoader(TensorDataset(x, y), batch_size=32, shuffle=True)

device = "cuda" if torch.cuda.is_available() else "cpu"
model = nn.Sequential(nn.Linear(128, 256), nn.ReLU(), nn.Linear(256, 10)).to(device)
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
criterion = nn.CrossEntropyLoss()

for epoch in range(3):
    for batch_x, batch_y in loader:
        batch_x = batch_x.to(device)
        batch_y = batch_y.to(device)

        optimizer.zero_grad()
        logits = model(batch_x)
        loss = criterion(logits, batch_y)
        loss.backward()
        optimizer.step()
```

你在这段代码里至少要看懂三件事：

1. 数据如何进入模型
2. 模型如何前向、反向和更新
3. 为什么 `.to(device)` 和 batch size 会直接影响性能

再往前走一步，就是分布式训练。

下面是一个 Horovod 风格的最小骨架：

```python
import torch
import horovod.torch as hvd

hvd.init()
torch.cuda.set_device(hvd.local_rank())

model = MyModel().cuda()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
optimizer = hvd.DistributedOptimizer(
    optimizer,
    named_parameters=model.named_parameters(),
)

hvd.broadcast_parameters(model.state_dict(), root_rank=0)
hvd.broadcast_optimizer_state(optimizer, root_rank=0)
```

看不懂也没关系，但你要先知道它想解决什么：

- 每个 worker 如何知道自己是谁
- 为什么所有 worker 要从同一组初始参数开始
- 为什么分布式训练会引入额外通信成本

## 第 8 章：推理系统和训练系统不是同一套目标

训练阶段最关心的是“把模型训好、训完、训得快”。

推理阶段最关心的是：

- 延迟低不低
- 每秒能处理多少请求
- 高峰期是否还能顶住
- 每个请求成本是否太贵
- 上线失败时能否回滚

### 推理系统最常见的 5 个优化方向

#### 1. Batching

把多个请求凑成一批一起算，提高吞吐。

#### 2. 模型优化

常见做法有量化、剪枝、蒸馏、算子融合、图优化。

#### 3. 推理运行时优化

例如 TensorRT、ONNX Runtime、Triton 这类工具，会帮助你把同一个模型跑得更快。

#### 4. 缓存

某些重复查询、特征、embedding 或 prompt 结果可以缓存，减少重复计算。

#### 5. 弹性扩缩容

流量波动时，系统要能自动加实例或减实例，控制 SLO 和成本。

### 推理系统的四个关键指标

- `p95/p99 延迟`
- `QPS / TPS`
- `错误率`
- `单位请求成本`

如果一个系统吞吐很高，但 p99 延迟很差，用户体验仍然可能很糟。

## 第 9 章：最小模型服务是什么样

学推理最重要的一步，是把模型真正暴露成服务。

例如 TensorFlow Serving 的部署思路通常是：

```bash
docker run -p 8501:8501 \
  --mount type=bind,source=/models/my_model,target=/models/my_model \
  -e MODEL_NAME=my_model \
  tensorflow/serving
```

然后通过 HTTP 接口请求：

```bash
curl -X POST http://localhost:8501/v1/models/my_model:predict \
  -H "Content-Type: application/json" \
  -d '{"instances": [[1.0, 2.0, 3.0, 4.0]]}'
```

这一步的意义不在于记命令，而在于建立一个认知：

**模型上线之后，不再是 notebook 里的对象，而是一个需要被版本化、部署、扩缩容、监控的服务。**

## 第 10 章：MLOps 为什么不是“多装几个工具”

MLOps 的本质，是把模型开发从“个人实验”变成“团队可持续工程”。

### MLOps 最少要解决的事情

1. 实验记录
2. 数据版本管理
3. 模型版本管理
4. 流水线自动化
5. 部署与回滚流程
6. 线上监控与反馈

### MLflow 的学习价值

MLflow 非常适合入门，因为它先解决最容易被忽略的一件事：**实验可追踪。**

```python
import mlflow

with mlflow.start_run():
    mlflow.log_param("lr", 1e-3)
    mlflow.log_metric("loss", 0.42)
```

你以后要能回答：

- 这个线上模型是哪次实验来的？
- 它用了什么超参数？
- 为什么这个版本比上个版本好？

### DVC 的学习价值

DVC 的价值在于把数据也纳入工程流程。

```bash
dvc init
dvc add data/raw
```

重点不是命令，而是观念：

**模型不是唯一需要版本控制的东西，数据和 pipeline 同样需要。**

### Kubeflow、Airflow、Ray 什么时候上

当你的训练、数据处理、部署已经不是单机脚本能管理时，就会开始需要工作流编排和平台化能力。

## 第 11 章：真实 AI 平台为什么长成那样

你学到这里，如果只记工具名，其实还不够。更重要的是看看真实生产平台为什么会那样设计。

### Google TFX / Kubeflow

这类平台的价值在于把数据验证、转换、训练、评估、上线、监控放进一条标准流程里。核心思路是：**把 ML 生命周期平台化。**

### Uber Michelangelo

Michelangelo 是很经典的企业 ML 平台案例。它让不同团队可以共享数据、训练模型、部署模型、监控预测。核心思路是：**让机器学习能力像平台服务一样被复用。**

### Meta 大规模 AI 集群

公开资料里，Meta 的 AI 集群核心特点是：

- 海量 GPU
- 高速网络互联
- 大规模存储系统
- 专门为训练和推理设计的基础设施

核心启发是：**大模型系统的瓶颈一定不是单一部件，而是整个系统链路。**

### Kubeflow / TFX / Michelangelo 这类案例带来的共同结论

1. 企业不会只靠几个脚本跑 AI。
2. 数据、训练、服务和监控必须是打通的。
3. 平台化的目标是降低重复劳动，提高复用性和治理能力。

## 第 12 章：AI Infra 到生产环境时，难点通常不在模型

很多团队在 demo 阶段都觉得进展很快，但一到生产环境就开始卡住。原因通常不是模型“不会写”，而是系统问题集中爆发了。

### 生产环境最常见的 6 类难题

#### 1. 数据质量

训练数据和线上数据分布不一致，模型效果就会掉。

#### 2. 成本

GPU、存储、流量、日志、特征服务，一起上来后开销会迅速变大。

#### 3. 监控

你不能只看 CPU 和内存，还要看模型效果、漂移、错误分布、尾延迟。

#### 4. 安全与权限

模型、数据、向量库、服务接口都可能成为攻击面。

#### 5. 合规与治理

企业会关心谁访问了什么数据、模型如何发布、是否满足审计要求。

#### 6. 人才与组织能力

很多报告都在强调一件事：企业缺的不只是机器，更缺能把 AI Infra 真正管起来的人。

## 第 13 章：一套够用的监控框架

你可以把 AI 系统监控分成四层。

### 系统层

- CPU
- GPU 利用率
- 显存
- 磁盘吞吐
- 网络带宽

### 服务层

- 请求量
- p95/p99 延迟
- 错误率
- 队列长度

### 模型层

- 准确率
- 召回率
- drift
- 输出分布异常

### 业务层

- 转化率
- 用户留存
- 推荐命中
- 投诉与异常反馈

真正成熟的 AI Infra，不是“模型部署上线了”，而是“模型上线后也能被持续观察和修正”。

## 第 14 章：该怎么开始学，才不会变成只收藏链接

你现在最不需要做的事，是继续加收藏夹。

你最需要做的，是按下面这个顺序推进。

### 第一步：把这份手册读完

重点不是记所有名词，而是你能不能讲清：

- AI Infra 的完整链路是什么
- 训练和推理为什么不同
- 计算、存储、网络、MLOps 各自解决什么问题

### 第二步：做 4 个最小实验

1. 跑单 GPU 训练
2. 跑一个分布式训练骨架
3. 部署一个最小模型服务
4. 用 MLflow 或 DVC 记录一次实验

### 第三步：开始看真实平台案例

这时候再看平台和企业案例，你就不会只看到“很多名词”，而会看到系统设计背后的原因。

### 第四步：做一个小闭环项目

最好的入门项目不是训练一个很强的模型，而是做一个小但完整的链路：

- 数据准备
- 模型训练
- 模型导出
- 服务部署
- 监控记录

只要你能把这五步串起来，你就已经从“资料阅读者”开始变成“AI Infra 实践者”了。
