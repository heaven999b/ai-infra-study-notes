## 这两篇大概率在讲什么

### 文章 1：`做Agent前，请先把Stanford CS336过一遍`

这篇文章的核心意思，大概率不是“先别做 Agent”，而是：

- 别一上来只学工具调用和工作流编排
- 先把 LLM 从数据到训练、从 tokenizer 到 post-training 的主干链路补起来
- 这样你做 Agent 时，很多现象不是“调参玄学”，而是能回到底层机制理解

也就是说，这篇更像是在讲一个原则：

`Agent 能力 = 底层模型理解 + 外层控制设计 + 工具/环境工程`

如果底层这一段是空的，后面很容易只会“接壳子”。

### 文章 2：`一天搞懂 Harness：我的 Codex 实战记录`

这篇文章的核心意思，大概率是在讲：

- 光会“用模型”还不够
- 真正让 Agent 稳定工作的，是模型外面的那层 harness
- Harness 不是零散脚本，而是一个完整的控制平面
- Codex、自定义 skill、验证器、状态管理、轨迹记录，这些组合起来才叫 agent runtime

也就是说，这篇更像是在讲一个视角切换：

从“我在向模型提问”，切换成“我在设计一个可执行、可验证、可复盘的 agent system”。

## 你可以从文章 1 学到什么

结合你本机已有的 `Stanford CS336` 作业仓库，这篇最值得学的不是“某个结论”，而是一整条知识主线。

### 1. Tokenizer 不是小细节

你会学到：

- 文本是怎么被切成 token 的
- BPE 为什么会影响压缩率、上下文利用和后续训练
- 为什么 tokenizer 设计会反过来影响模型行为

本机对应材料：

- `stanford-cs336-spring2025-assignments/assignment1-basics`

### 2. Transformer 和训练循环才是 Agent 的地基

你会学到：

- Transformer LM 的基本结构
- cross-entropy、AdamW、训练循环、checkpoint 的基本实现
- 为什么 loss、perplexity、batch、lr 这些指标不是“调参黑箱”

这部分的意义是：

- 以后你在看模型输出异常、长上下文退化、工具调用不稳时，脑子里会有底层解释框架

### 3. Systems 能力决定你是不是只会“调 API”

你会学到：

- profiling / benchmarking harness
- FlashAttention 这类系统优化为什么重要
- DDP、optimizer sharding 这些训练系统概念在干什么

这会直接提升你对“模型为什么慢、为什么贵、为什么扩不起来”的判断力。

### 4. Scaling、Data、Alignment 是 Agent 之后必然会遇到的问题

你会学到：

- scaling laws 是怎么估算算力和模型规模关系的
- 数据清洗、去重、PII 处理为什么会影响最终能力
- SFT、Expert Iteration、GRPO 这些 post-training / alignment 方法是在解决什么问题

这部分很重要，因为现在很多 Agent 问题表面上像 workflow 问题，实际上是：

- 数据分布问题
- reward / verification 设计问题
- post-training 问题

## 你可以从文章 2 学到什么

结合你本机已有的 `Agentic-RL-harness` 和 `agent-harness-playground`，这篇最值得学的是 harness 思维。

### 1. Harness 是模型外面的控制平面

你需要建立的核心概念是：

- 模型负责生成候选动作
- harness 负责定义环境、工具、状态、步骤推进、验证、观测和约束

在 `agent-harness-playground` 里，可以把 harness 理解成：

- environment access
- tools
- context
- state
- lifecycle
- verification
- observability
- governance

这套东西决定了一个系统是不是“像 agent 一样工作”。

### 2. 评估 Agent 时，不能把模型和 harness 混成一个东西

`Agentic-RL-harness` 这套材料最重要的启发是：

- 固定底层 LLM
- 单独优化 outer-loop harness
- 用 reward 和 process metrics 看控制策略是不是更成熟

这比只看最终答对没答对，要更接近真正的 agent engineering。

### 3. 好的 Agent 不是一次答对，而是会搜索、检查、修订、提交

这类 harness 材料强调的是：

- 不是只看 final answer
- 还要看过程是否成熟
- 是否有验证步骤
- 是否会过早提交
- 是否能在预算内做出更稳的决策

这对你之后做：

- Codex workflow
- skill 设计
- benchmark 设计
- verifier / judge 设计

都会非常有帮助。
