# 08 中文开源教程与实战资源补充

这一页补的是另一类很有价值的资料：

- 不是偏底层 AI Infra 的通识框架
- 而是中文社区里更系统、更容易上手、也更贴近实操的 Agent / LLM / RAG / 微调 / 部署 / Vibe Coding 教程

如果说根目录前几份笔记是在帮你建立总地图，这一页更像是在回答：

- 现在想学 Agent，先看哪套中文教程
- 想补 RAG、微调、部署，哪几套最顺手
- 想从“看概念”走到“自己跑起来”，哪些资源更值得优先投入

## 为什么放在这个仓库

这批链接更适合放在 `ai-infra-study-work`，而不是 `ai-infra-learning` 或个人主页，原因很简单：

- `ai-infra-study-work` 本来就是“学习导航 + 上游资料索引”
- 这批资源大多是外部教程、外部仓库、外部文档
- 它们更适合作为学习路线里的“入口页”，而不是写进一份单体教材

也就是说，这里适合先做“导航和取舍”，以后如果你真的要长期啃某个项目，再考虑把它做成 `upstream-learning-sources` 里的快照。

## A. Datawhale 这批教程怎么放进你的学习路线

### 1. [hello-agents](https://github.com/datawhalechina/hello-agents)

这是这批里最适合放在前排的 Agent 教程。

适合你现在的原因：

- 它是系统化教程，不只是零散 demo
- 它把 Agent 原理、范式、框架、记忆、评估、Agentic RL 串成了一条线
- 它和你本机已有的 `Agentic-RL-harness`、`agent-harness-playground` 能形成互补

怎么用最值：

- 把它当“中文总览教程”
- 把本机两个 harness 仓库当“代码侧验证材料”

### 2. [happy-llm](https://github.com/datawhalechina/happy-llm)

这是比较适合补 LLM 主干概念的入口。

它最适合承担的角色是：

- 给 Agent 学习补模型地基
- 帮你把预训练、微调、推理这些主线重新串起来
- 让你之后看 post-training、reward、tool use 时没那么悬空

如果你现在正在补 `Stanford CS336`，这份可以作为中文辅助材料，而不是替代品。

### 3. [all-in-rag](https://github.com/datawhalechina/all-in-rag)

这份最适合放进你“应用层工程理解”的路线里。

它的价值不是只会做一个问答 demo，而是：

- 帮你把检索、索引、chunking、召回、生成、评测这些环节串起来
- 让你从系统角度看 RAG，而不是只记几个框架名

它和 `hello-agents` 的关系是：

- `hello-agents` 偏 Agent 视角
- `all-in-rag` 偏知识增强与应用工程视角

### 4. [self-llm](https://github.com/datawhalechina/self-llm)

这份非常适合放在“从教程走向部署和微调”的位置。

它最有用的地方在于：

- 更贴近中文用户常见环境
- 强调 Linux 下的大模型微调与部署
- 对“跑起来”这件事更友好

如果你后面要真机做微调或部署，这份会比纯概念教程更快把你拉到工程侧。

### 5. [vibe-vibe](https://github.com/datawhalechina/vibe-vibe) 和 [easy-vibe](https://github.com/datawhalechina/easy-vibe)

这两份更适合单独看成一条“AI 编程 / AI 应用搭建”支线。

更建议这样理解：

- `vibe-vibe` 更像系统化课程入口
- `easy-vibe` 更像更轻、更友好的零基础实战入口

它们和主线 AI Infra 不完全重合，但对你理解“现在普通用户和开发者到底怎么用 AI 快速做东西”很有帮助。

如果你以后想做：

- 面向初学者的教程整理
- AI coding 工作流总结
- 从 prompt 到应用落地的示例库

这两份很值得保留在导航里。

### 6. [base-llm](https://github.com/datawhalechina/base-llm)

这份更偏“从 NLP 到 LLM 的算法连续体”。

它的价值在于：

- 帮你把经典 NLP 和现代 LLM 接起来
- 适合补算法直觉
- 适合在你想把“会用模型”升级成“能解释模型与任务关系”时回看

### 7. [every-embodied](https://github.com/datawhalechina/every-embodied)

这份建议单独标成“具身智能支线”。

它和你当前的主线不完全重合，但很适合当未来扩展方向保留。

这里顺手记一下：

- 你给的仓库名里多了一个 `we`
- 仓库实际是 `every-embodied`

## B. 工具链与训练实战资源

### 1. [Unsloth 教程合集](https://docs.unsloth.ai/basics/tutorials-how-to-fine-tune-and-run-llms)

这套资料更适合放在“具体模型怎么调、怎么跑、怎么省资源”的位置。

它的特点是：

- 对单模型教程覆盖很全
- 更偏实操参数、显存、运行方式、常见坑
- 很适合当“动手时查手册”

对你来说，它不像主教材，更像高价值工程参考。

### 2. [Unsloth RL Guide](https://docs.unsloth.ai/basics/reinforcement-learning-guide)

这份更适合放在 `hello-agents` 和 `Agentic-RL-harness` 之后看。

原因是：

- 它偏 RL / post-training / GRPO / reward 设计
- 更像“开始认真碰 agent training 时的工程说明书”

如果你前面还没建立好：

- SFT 是什么
- preference / reward / rollout 在干什么
- harness 和 training 的边界在哪里

那它会显得有点跳。

### 3. [ms-swift](https://github.com/modelscope/ms-swift)

这份建议你在仓里标成“国产一站式训练/推理/评测框架”。

它特别适合承担的角色是：

- 当你想快速评估一个训练/微调框架值不值得装
- 当你想看国内生态里更完整的一站式方案
- 当你需要把训练、推理、量化、部署放在一个工具链里理解

按我在 `2026-06-30` 看到的仓库首页描述，它现在已经强调支持 `600+` 纯文本大模型和 `400+` 多模态大模型，比你原文里的 `500+` / `200+` 更新了一截。

### 4. [LLaMA-Factory](https://github.com/hiyouga/LlamaFactory)

这份适合放在“通用微调工具链代表项目”这一栏。

它的使用价值很直接：

- 社区认可度高
- 支持模型面广
- 微调、偏好优化、实验跟踪这些环节比较完整

如果你后面只想真跑一个公开、成熟、文档也比较完整的微调框架，它常常会是比“自己搭一堆脚本”更省力的起点。

它会是一个很好的试验性项目。

## D. 我建议你现在怎么取舍

如果你的目标是 **继续补 Agent 主线**，优先顺序建议是：

1. `hello-agents`
2. `happy-llm`
3. `Unsloth RL Guide`
4. `ms-swift` 或 `LLaMA-Factory`

如果你的目标是 **补 RAG 和应用工程**，优先顺序建议是：

1. `all-in-rag`
2. `hello-agents` 里的记忆 / 检索部分
3. `self-llm`

如果你的目标是 **尽快做可运行项目**，优先顺序建议是：

1. `self-llm`
2. `easy-vibe`
3. `LLaMA-Factory`

如果你的目标是 **保留未来扩展方向**，先挂着别急着深挖的是：

- `every-embodied`
- `lanshu-animated-architecture-diagram`

## E. 以后要不要继续往 `upstream-learning-sources` 里丢

我建议现在先不要一股脑全量同步。

更好的做法是：

- 先把这页当“导航页”
- 真正决定长期深挖的，再做精选快照

如果后面只挑第一批值得做快照的，我会优先选：

- `hello-agents`
- `all-in-rag`
- `self-llm`
- `ms-swift`

原因是这四个最接近你当前学习主线，也最容易跟你现有本地仓库形成联动。
