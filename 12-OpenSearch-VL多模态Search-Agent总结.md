## 一句话先说清它在做什么

`OpenSearch-VL` 做的不是一个“会看图回答问题”的普通 VLM。

它做的是：

- 一个能围绕图像做多轮调查的多模态搜索 agent

也就是说，模型不是单次前向直接答题，而是会：

- 看图
- 裁剪局部
- 增强图像
- 做文本 / 图像 / 网页搜索
- 访问页面
- 再回头汇总答案

官方 README 用的词很准确：

- `closed loop`

这点非常关键，因为它决定了整套工作重心不只是模型本身，而是：

- 数据
- 工具环境
- 轨迹
- RL 稳定性

## 一、数据合成：它最强的地方之一

你给的 8 步数据合成链，其实就是这套工作的地基。

如果压缩成一句话，它的目标是：

- 故意构造那种单跳搜一下解决不了、必须多跳、多工具、多轮搜索才能解开的视觉问答样本

### 1. 维基百科路径采样，不是随便凑知识点

这里的关键设计是：

- 把维基百科看成有向图
- 从种子实体出发做 `2-4` 跳受限随机游走
- 跳过消歧义页、列表页、循环路径和高入度枢纽节点

这一步的价值是：

- 防止样本都退化成“热门实体 + 直接属性匹配”

同时它给节点分配：

- 锚点
- 桥接点
- 答案点

这样后续问题生成就不是随意拼接，而是有明确结构角色。

### 2. 先生成标准问答，再做模糊化

这里的设计顺序很聪明：

1. 先从答案节点抽出短而唯一的答案。
2. 再用强模型生成标准问题，确保问题只指向那个答案属性。
3. 然后再从最远桥接点往回，逐步把实体名改写成属性描述。

这一步的目的不是把题目写得“花”，而是去掉：

- 实体名直接泄露
- 一眼就能搜到的 shortcut

也就是你提到的那三个关键约束：

- 答案不变
- 唯一指向
- 无实体泄露

### 3. 视觉锚定，让题真正变成多模态

这是这套数据比普通多跳 QA 更厉害的地方。

它不是只把文字问题绕复杂，而是：

- 给锚点实体去 Wikimedia Commons 找图
- 用 `CLIP` 选出匹配图像
- 再把问题里的锚点名替换成视觉指代表达

这会把问题从：

- 文本检索题

变成：

- 真正要“看图识别锚点，再开始搜索”的题

### 4. 两阶段过滤，专门踢掉“其实不用多工具”的样本

这个步骤很关键。

很多看起来复杂的数据，其实最后都被模型用一个 shortcut 解掉了。

这套方法显式做了两轮过滤：

- 先用无工具模型过滤
- 再用单轮图像搜索过滤

只保留那种：

- 必须多工具
- 必须多跳
- 必须走过程

才能解决的题

这一步直接提高了轨迹学习的价值密度。

### 5. 数据增强不是锦上添花，而是把“图像修复工具”做进训练分布

你提到的第 7 步很值得单独强调：

- 抽 `10%` 样本做模糊、降采样、透视畸变
- 再配对锐化、超分、透视校正工具

这意味着图像增强工具不是只在推理时外挂，而是：

- 在训练数据里就被明确配上了使用场景

这样模型学到的不是“知道有这些工具”，而是：

- 知道什么时候该用它们

### 6. 专家轨迹合成，才是把数据变成 agent 训练资产的那一步

这里最关键的是：

- 在真实工具环境里生成轨迹
- 做答案正确性 + 过程级双重拒绝采样

最终得到：

- `36,592` 条 SFT 轨迹
- 再抽 `8k` 独立样本做 RL

官方 README 当前也对应给出了两个公开数据集名字：

- `SearchVL-SFT-36k`
- `SearchVL-RL-8k`

## 二、工具环境：这不是“会调几个 API”，而是一个统一 agent runtime

从官方 README 当前可见内容看，仓库把工具分成：

- 检索工具 `T_r`
- 视觉工具 `T_v`

并强调：

- SFT
- RL
- 推理

三段共享同一套工具环境。

### 1. 官方 README 当前列出的统一工具集

GitHub README 当前可见的共享工具包括：

- `crop`
- `layout_parsing`
- `text_search`
- `image_search`
- `web_search`
- `visit`
- `perspective_correct`
- `super_resolution`
- `sharpen`
- `python_interpreter`

你给的中文拆解里，重点强调的是其中最核心的几类：

- `TEXTSEARCH`
- `IMAGESEARCH`
- `SHARPEN`
- `SUPERRESOLUTION`
- `PERSPECTIVECORRECT`
- `CROP`
- `OCR`

如果把两边对齐来看，可以这样理解：

- 你这份拆解是在讲核心业务能力
- 官方 README 则把它们扩成了完整 runtime 接口

### 2. 检索工具的关键，不只是搜，而是搜完还要结构化返回

你这边拆得很细：

- 文本搜索 = `Serper` + `JINA` 网页解析 + `Qwen3-32B` 摘要
- 图像搜索 = `PolarisLens` 反向图搜 + 实体识别

这说明它不是把搜索引擎原始页面直接喂回模型，而是做了：

- 搜索
- 解析
- 摘要 / 结构化观察

这对于多轮 agent 很关键，因为模型真正需要的是：

- 可操作的 observation

而不是一大坨原始页面噪音。

### 3. 图像增强工具不是装饰，而是 Search Agent 的前置恢复层

这里最值得学的是：

- 锐化：`OpenCV` 反锐化掩码
- 超分：`EDSR`
- 透视校正：`Canny` 边缘 + 四边形拟合 + 透视变换

这说明他们把视觉工具设计成：

- 为搜索成功率服务

也就是：

- 先修图
- 再识别
- 再搜

而不是把视觉增强当单独任务。

### 4. 裁剪和 OCR 是“注意力控制器”

这组工具非常像 agent 的局部视觉注意力系统。

- `crop` 负责把局部区域单独拿出来
- `OCR` / `layout_parsing` 负责从局部区域抽文本

这一步很关键，因为很多搜索题真正难的不是：

- 不会搜

而是：

- 根本没把图里关键区域抠出来

## 三、SFT：它在教模型学会“怎么走过程”

SFT 这部分，最重要的不是参数量，而是训练目标。

它在学的不是最终答案本身，而是：

- 多轮推理迹
- 工具调用节奏
- 图像观测和文本观测的交错使用

### 1. 这是全参数多模态微调，不是轻量补丁

你给的摘要里提到：

- 基座包括 `Qwen3-VL-8B / 30B-A3B / 32B`
- 视觉塔和多模态投影器都不冻结

官方 README 当前也把 SFT 描述为：

- `full-parameter fine-tune of LLM + ViT + projector`

这意味着它的目标不是：

- 在原模型表面贴一个 agent 外壳

而是：

- 让多模态底座本身就更适合工具化、多轮搜索式行为

### 2. 掩码策略也说明它关注“谁在说话”

你给的关键信息里提到：

- 观测 token 被掩码
- 只对模型生成 token 计算损失

这个设计很合理，因为训练目标不是去复述观察，而是学会：

- 如何基于观察做推理和调用工具

### 3. 训练规模说明这不是小实验

你给的 SFT 细节里提到了：

- `36,592` 条多轮专家轨迹
- 最大长度 `32,000 token`
- `DeepSpeed ZeRO-3`
- 学习率 `2e-5`
- 余弦调度
- 训练 `8` 轮

这已经是很标准的“严肃 agent SFT”规模了。

## 四、RL：真正有意思的是 fatal-aware GRPO

这部分是整套工作的算法亮点。

普通多轮工具 RL 一个很大的问题是：

- 前半段明明走得挺对
- 后半段连续工具报错把整条轨迹拖死
- 如果整条一起惩罚，模型会把前面有效部分也学坏

### 1. 他们先显式定义了“致命错误”

你的摘要里给出的规则是：

- 连续 `3` 次工具执行错误

官方 README 当前也把 fatal threshold 写成：

- `3 consecutive tool-execution errors`

这一步很重要，因为它把“崩了”从模糊感受变成了：

- 可检测事件

### 2. 致命感知掩码，本质上是在保护有效前缀

核心做法是：

- 找到崩溃起点
- 把之后所有无效 token 置零
- 只保留崩溃前有效内容参与梯度

这比“整条轨迹一起扣分”精细得多。

它解决的是：

- 多轮 agent 中后期事故不该反向污染前期正确推理

### 3. 单边优势钳制，是另一个很聪明的保护机制

你提到：

- 非致命轨迹用标准归一化优势
- 致命轨迹取 `max(归一化奖励, 0)`

这个设计的直觉是：

- 致命轨迹不继续给坏后缀加负信用
- 但也不否定前缀里那些本来做对的部分

这和 fatal-aware mask 组合起来，才让多轮 RL 更稳定。

### 4. 奖励不是单一正确率，而是复合奖励

你的摘要里给出的结构是：

- 格式奖励 `r_fmt`
- 准确率奖励 `r_acc`
- 查询质量奖励 `r_query`
- 用 `alpha = 0.8` 做融合

官方 README 当前也明确写出了复合奖励，且 `query` 质量被单独纳入。

这说明他们并不满足于：

- 最后答对就行

而是进一步要求：

- 查询行为本身要像样

这对 Search Agent 很关键。

## 五、最值得你拿走的 5 个方法论点

### 1. 先把 shortcut 打掉，再做 agent 数据

如果题本身一跳就能解，再复杂的 agent 训练也会学歪。

### 2. 让图像增强工具出现在训练数据里，而不是只出现在推理菜单里

工具不是“可用列表”，而是“被场景约束过的行为选项”。

### 3. SFT 的核心不是答案，而是高质量轨迹

真正训练的是：

- 调查方式
- 工具节奏
- 失败恢复前的行为模式

### 4. RL 的重点不是奖励函数花样，而是如何处理长轨迹崩溃

fatal-aware mask 和 one-sided advantage clamping 解决的是很真实的 agent 训练痛点。

### 5. 工具环境必须在数据合成、SFT、RL、推理之间保持一致

不一致的话，模型学到的是：

- 训练时能做
- 推理时不会做

而 `OpenSearch-VL` 的一个长处，就是它把这四段链路接在了同一个环境里。
